# Domain Research: product（产品运营类）

## 1. 風格核心定義

**本質目標**: プロダクトの機能・特性・操作方法を、ユーザーが理解→活用→定着するための情報として最適化すること。「何ができるか」「どう使うか」「なぜこの製品を選ぶか」を伝えるコンテンツ全般。

**他風格との核心区別**:
- vs `marketing`（営業推広）: marketing は「行動させる」（CTA必須、ベネフィット先行、PASONA構造）。product は「理解させる」（Feature→Benefit構造、CTA任意、Logos主導）。marketing がコンバージョンを追求するのに対し、product はコンプリヘンション（理解完了）を追求する。
- vs `news`（新闻事件类）: news は時事イベントの事実報道。product は特定プロダクトの恒常的な特性説明で、時間依存性が低い。
- vs `support`（客服支援）: support は問題解決（Q&A・手続き案内）に特化し謙譲語IIを使用。product は機能説明・教育が主目的で、標準的なです/ます体。
- vs `legal`（法律合規）: legal は法的義務/免責の記述。product はユーザー体験向上のための情報提供で、promotional tone は控えめながら「活用できる」等のポジティブ表現を許容。

## 2. 目標受众与触达时机

| 受众セグメント | 触達タイミング | 期待される行動 |
|---|---|---|
| 新規口座開設検討者 | プラットフォーム比較段階 | 機能理解→差別化認知→選択意思決定 |
| 新規開設直後ユーザー | オンボーディング期間（開設後1–14日） | 機能把握→初回取引→習慣化 |
| 既存ユーザー（機能未活用） | 新機能リリース時、利用率低下時 | 新機能認知→試用→定着 |
| アクティブトレーダー | 高度機能リリース時 | 高度ツール活用→取引効率化→LTV向上 |
| 潜在クロスセル対象 | 新商品カテゴリ訴求時 | 商品理解→新カテゴリ取引開始 |

**触達チャネル**: 商品説明ページ、ヘルプセンター、アプリ内チュートリアル、機能紹介動画、リリースノート、比較表ページ、オンボーディングメール

## 3. 語体/レジスタ特征

| 要素 | 仕様 |
|------|------|
| 動詞詞尾 | です/ます体（全 scene 統一） |
| 敬語等級 | 丁寧語レベル（ございます/承ります不要）; 「〜できます」「〜いただけます」が標準 |
| 句長 | 短文案: 15–25字; 詳細説明: 20–35字; マニュアル: 20–40字; スクリプト: ≤30字 |
| 語気 | 説明的・教育的; 適度にポジティブ（「〜できます」「〜を活用できます」）だが誇張禁止 |
| 構造原則 | Feature → Benefit 必須ペア; 機能名提示→ユーザーメリット展開 |
| 専門用語 | 初出は括弧説明付き「〜（○○とも呼ばれる手法）」; 2回目以降は素の形 |
| 禁止表現 | 煽り（「今すぐ」「チャンス」「！」）、断定的優位性（「業界No.1」※根拠なし）、hedging不要（事実機能の説明なので推測不要） |
| 二人称 | 「お客様」または無主語構文; 「あなた」は原則不使用 |

## 4. 日本金融行業の典型載体

| 媒介/フォーマット | 参考ソース | 特徴 |
|---|---|---|
| 商品説明ページ | SBI証券「国内株式」「投資信託」、楽天証券「商品・サービス」 | 機能列挙+各ベネフィット説明、比較表あり |
| 機能紹介LP | moomoo証券「機能紹介」、マネックス証券「ツール」 | スクリーンショット+短文説明の組み合わせ |
| リリースノート/更新情報 | 各社アプリストア更新履歴、お知らせページ | バージョン番号+変更点リスト+影響説明 |
| ヘルプセンター/マニュアル | 楽天証券「使い方ガイド」、SBI証券「操作ガイド」 | 手順番号付き、スクリーンショット付き |
| オンボーディングメール | 各証券会社 開設完了後メールシリーズ | ステップバイステップ誘導、機能ハイライト |
| 動画チュートリアル | moomoo証券YouTube、楽天証券「使い方動画」 | ナレーション+画面録画、≤30字/文の口語体 |
| 比較表/USPカード | 各社「他社比較」ページ、広告バナー | 数値/機能の対比表、3–5項目に絞った差別化ポイント |
| アプリ内ツールチップ/コーチマーク | 各社アプリUI | ≤50字の極短説明、初回表示→非表示切替 |

## 5. 推薦 Tier 2 子場景

| Scene ID | 名称 | 説明 | 典型字数 |
|---|---|---|---|
| `short` | 短産品文案・USPカード | 単一セリングポイントの極短説明。差別化が命。 | 30–80字 |
| `detail` | 詳細説明ページ | 全機能のFeature→Benefit展開。スキャン可能な構造。 | 200–500字/セクション |
| `manual` | 使用説明・手順書 | 操作手順を番号付きステップで記述。1アクション/1ステップ。 | 150–400字 |
| `script` | 動画・音声スクリプト | 口語体での機能説明。1文≤30字、書き言葉排除。 | 200–600字 |
| `release` | リリースノート・更新情報 | バージョン更新内容を構造的に列挙。変更→影響→手順。 | 100–300字 |
| `onboard` | オンボーディング | 新規ユーザー向け初期設定・初回利用ガイド。ステップ進行+達成確認。 | 200–500字 |
| `compare` | 比較表・USP対比 | 競合/旧バージョンとの機能比較。表形式+補足1–2文。 | 100–200字（表除く） |

## 6. 関鍵約束草案

| # | 約束 | 根拠 |
|---|------|------|
| PC-1 | **Feature→Benefit 必須ペア**: 機能記述には必ず対応するユーザーメリットを付す。機能名のみの羅列は禁止。 | 「何ができるか」だけでなく「なぜ嬉しいか」を伝える |
| PC-2 | **専門用語初出説明**: 金融/技術用語の初出時に括弧内で平易な説明を付す。2回目以降は不要。 | 初心者〜中級者が対象読者に含まれるため |
| PC-3 | **誇張禁止・根拠必須**: 「業界最安」「No.1」等の優位性主張には具体的根拠（出典、数値、条件）を併記。根拠なき断定禁止。 | 金融庁ガイドライン/景品表示法準拠 |
| PC-4 | **CTA 任意・控えめ**: CTA は任意。付す場合は低コミットメント形（「詳しくはこちら」「試してみる」）。「今すぐ申し込む」等の強い CTA は marketing 領域。 | product は理解促進が目的で、コンバージョン圧力は抑制 |
| PC-5 | **1アクション/1ステップ（manual/onboard）**: 手順記述では1ステップに複合動作を含めない。UI要素名は「」括弧で囲む。 | ゼロバックトラック原則：迷わず完了できること |
| PC-6 | **構造的スキャナビリティ**: 3項目以上の並列情報は箇条書き/表形式に構造化。4文以上の連続散文ブロック禁止（manual/detail）。 | Web/アプリでの閲覧を前提としたUI最適化 |
| PC-7 | **能動態優先**: 受動態（〜されます/〜となっています）を能動態に変換。ただし manual で主語を隠すための受動態（「表示されます」）は許容。 | 読みやすさ + product 文体の明快さ |
| PC-8 | **数値具体化**: 曖昧表現（「高速」「簡単」「多彩」）を可能な限り具体化（「約3秒で完了」「3ステップで注文」「15種類のチャート」）。入力に数値がある場合は必ず保持。 | Logos 主導：具体性が信頼性を生む |

## 7. 隣接風格との境界

### product vs marketing（営業推広）

| 軸 | product | marketing |
|---|---|---|
| 目的 | 理解・活用促進 | 行動変容・コンバージョン |
| CTA | 任意（低コミットメント） | 必須（行動動詞） |
| 構造 | Feature → Benefit | Benefit → 社会的証明 → CTA |
| 語気 | 説明的・教育的 | 訴求的・能動的 |
| 煽り表現 | 禁止 | 許容（「今すぐ」「限定」等） |
| 判定基準 | 主目的が「理解させる」→product / 「行動させる」→marketing |

### product vs support（客服支援）

| 軸 | product | support |
|---|---|---|
| 起点 | プロアクティブ（機能紹介） | リアクティブ（問題解決） |
| 敬語 | 丁寧語（です/ます） | 謙譲語II（承ります/ございます） |
| 構造 | Feature→Benefit / Step-by-step | Q→A→チャネル案内 |
| 語彙 | 「〜できます」「〜いただけます」 | 「承ります」「ご案内いたします」 |
| 判定基準 | ユーザーが知りたくて来る（教育）→product / 困って来る（救済）→support |

### product vs news（新闻事件类）

| 軸 | product | news |
|---|---|---|
| 時間依存性 | 低（恒常的特性） | 高（時事イベント） |
| 主語 | プロダクト/機能 | 市場/企業/経済指標 |
| レジスタ | です/ます（ユーザー向け） | だ/である（客観報道） |
| 判定基準 | 特定プロダクトの機能/特性を説明→product / 市場イベント報道→news |

### product vs legal（法律合規）

| 軸 | product | legal |
|---|---|---|
| トーン | ポジティブ（活用促進） | ニュートラル/ネガティブ（リスク警告） |
| 表現 | 「〜できます」「〜を活用できます」 | 「〜のおそれがあります」「〜責任を負いません」 |
| CTA | 「試してみる」（任意） | 禁止 |
| 判定基準 | ユーザーメリットを説明→product / 義務・免責・リスクを説明→legal |

## Open Questions

- `release`（リリースノート）は product に属するか news に属するか？→ 提案：プロダクト機能の変更点説明がメインなので product。市場サービス全般の告知は press（report T1）が適切。
- `compare`（比較表）と marketing/lp の差別化：比較表は情報提供が主目的（Logos）だが、LP内に埋め込まれる場合は marketing の文脈に従う。独立ページの場合は product/compare。
- `onboard` と support/guide の境界：onboard は「初回設定の成功体験」を目的とするプロアクティブコンテンツ。guide は「手続きで困った時」のリアクティブコンテンツ。起点の違いで分離。
- 既存 `short`/`detail`/`manual`/`script` との関係：これらは現行の style-guide.md で定義済み。`release`/`onboard`/`compare` は新規追加候補。既存 scene は定義を引き継ぎつつ、上記約束草案で統一的な T1 制約を適用。
